Guardian — the Sign-Intent Flow

How a natural-language intent becomes a signed on-chain transaction, gated by a deterministic, fail-closed firewall (WYSIWYS).

Dewlock is an intent-firewall for agentic Sui DeFi. A single agent (holding zero user-fund keys) compiles a stated goal into one unsigned PTB; a deterministic Guardian re-derives the math and dry-runs that exact bundle; you sign the literal artifact you saw. The diagram below traces that path.

Figure 1 — Sign-intent flow. Any failing Guardian gate blocks before a signature is ever requested.

Guardian gate pipeline (executed in order)

  1. Allowlist — every PTB call {pkg::mod::fn} must be pre-approved (runs first).
  2. Action-shape — the MoveCall set must match exactly one actionType template ⇒ one value action per PTB.
  3. Coin-type provenance — coin types verified on-chain via CoinMetadata (by TYPE, never symbol).
  4. Injection provenance — a derived recipient (from inferred / recalled / pool data) forces a confirm gate.
  5. Trusted USD price — real oracle; no price ⇒ block (cannot value ⇒ cannot verify).
  6. Server caps — per-tx + per-day USD caps; invalid config ⇒ block everything.
  7. SuiNS lookalike — homoglyph-normalized edit-distance vs verified contacts.
  8. Min-out re-derive — swaps: recompute min-output from on-chain decimals + the same route source.
  9. Orderbook / Lending — POST_ONLY / self-match / expiry; lending is health-improving only.
  10. Dry-run + WYSIWYS digest — dry-run the exact bytes (fail-closed); approvedDigest = sha256(txBytes).
  11. Authoritative value gate — re-value from actual net outflow; block if outflow > 1.5× declared.
WYSIWYS (What You Sign Is What You Saw). The Guardian computes approvedDigest = sha256(txBytes) over the exact bytes it dry-ran and previewed. The wallet signs those same bytes; the sign hook asserts signed bytes == approved bytes. The agent never holds user-fund keys and the LLM never overrides a gate — code is the gate, fail-closed on every dependency.
