The moat, applied to intent. The LLM only proposes an action; a deterministic router re-derives the action category from the user's literal command and blocks any prepared trade whose actionType category does not match — catching LLM action-misdetection before a PTB ever reaches the user for signing. Same fail-closed posture as the Guardian.

1 · What is under test

Module packages/agent/src/intent/intent-router.ts (pure, deterministic, no LLM) exposes three units, wired into the agent stream at apps/web/app/api/agent/route.ts.

UnitResponsibilityReturns
routeAction(text)Classify the user's literal command into an action category from adapter keywords (swap / lend / stake / send) + guard-only verbs (bridge / limit). First value-verb of the first clause wins.category · or null (read-only / no verb)
categoryForActionType(at)Map a proposed prepareTrade actionType to its category. Composite & DeepBook-lifecycle return null (no single-verb mapping → not cross-checked).category · or null (skip)
assertActionMatchesText(text, at)The gate. Blocks only when both sides are non-null and differ. null on either side → pass (no false positive).{ ok, reason? }

Server wiring — where the gate fires

user text→ LLM proposes prepareTrade→ capture actionType from tool-call part→ on tool-result ok:true→ assertActionMatchesText
match →emit preview → user signs mismatch →emit BLOCK · gates:[intent_mismatch] → BlockCard
Why a hard block matters: a misrouted action that reaches the preview is one click from signing the wrong on-chain transaction. The gate replaces the signable preview with a block card carrying the same { ok:false, reasons, gates } shape the Guardian uses — so a misdetected trade can never be signed.

3.5 · Detailed walkthroughs

M-1 · Block path

  1. Type lend 100 USDC on navi; LLM (misdetecting) proposes prepareTrade(actionType:"swap").
  2. Expect: no signable swap preview — a BlockCard with "Refusing for safety: the prepared action (\"swap\") doesn't match what you asked (\"lend\")…", gate intent_mismatch.
  3. Done when: nothing signable shown; rephrase prompt; no wallet popup.

M-2 · Pass path

  1. Type swap 5 SUI to USDC; LLM proposes actionType:"swap".
  2. Done when: normal Guardian-checked swap preview renders; gate adds zero friction to a correct turn.

M-3 · No-false-positive

  1. show my portfolio → portfolio renders, gate silent (routeAction null).
  2. Atomic swap 5 SUI to USDC then lend it → composite skipped (categoryForActionType null); chain proceeds.
  3. Done when: neither legitimate flow is blocked.

Known limitation (fail-open, by design): the gate reads actionType from the Mastra tool-call stream part because the ok:true result carries no actionType. It relies on the tool-call part arriving before the tool-result in Mastra 1.42's fullStream (it does). If that ordering ever changes, the captured actionType is null and the gate fails open — safe (never blocks a legit turn) but blind. Hardening follow-up: add actionType to the prepareTrade output schema for a result-side fallback.
